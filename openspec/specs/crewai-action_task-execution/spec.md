# CrewAI 액션 작업 실행 명세

## Purpose
시스템이 `crewai-action` 유형의 작업을 주기적으로 감지하여, 작업 컨텍스트로 멀티 에이전트 크루를 구성·실행하고, 진행 상태와 최종 결과를 이벤트로 전달하도록 보장한다. 도구 연결 실패 같은 부분 장애가 있어도 입력 컨텍스트만 있으면 작업을 완료하고 자원을 정리한다.

## Requirements

### Requirement: 작업 폴링 및 수신
시스템은 `agent_type`이 `crewai-action`인 작업을 주기적으로 폴링하여 SHALL 수신한다. 폴링 주기는 5초이며, 새 작업이 감지되면 해당 작업의 실행 컨텍스트를 처리 대상으로 삼는다.

#### Scenario: 신규 작업 감지 및 처리 시작
- **GIVEN** 처리되지 않은 `crewai-action` 작업이 존재한다
- **WHEN** 폴링 주기가 도래한다
- **THEN** 시스템은 해당 작업을 수신하여 실행을 시작한다

#### Scenario: 처리할 작업이 없을 때 대기
- **GIVEN** 처리되지 않은 `crewai-action` 작업이 없다
- **WHEN** 폴링 주기가 도래한다
- **THEN** 시스템은 작업을 실행하지 않고 다음 폴링 주기까지 대기한다

### Requirement: 작업 컨텍스트에서 실행 정보 추출
시스템은 수신한 작업 컨텍스트에서 실행에 필요한 식별자와 입력값을 SHALL 추출한다. 프로세스 인스턴스 식별자는 `root_proc_inst_id`를 우선 사용하고, 없으면 `proc_inst_id`를 사용한다. 작업 식별자(`task_id`), 테넌트 식별자(`tenant_id`), 폼 식별자(`form_id`)도 컨텍스트에서 해소한다.

#### Scenario: 루트 인스턴스 식별자 우선 해소
- **GIVEN** 작업 컨텍스트의 `row`에 `root_proc_inst_id`와 `proc_inst_id`가 모두 존재한다
- **WHEN** 시스템이 실행 컨텍스트를 초기화한다
- **THEN** 프로세스 인스턴스 식별자는 `root_proc_inst_id` 값으로 해소된다

#### Scenario: 루트 인스턴스 식별자 부재 시 대체 해소
- **GIVEN** 작업 컨텍스트의 `row`에 `root_proc_inst_id`가 없고 `proc_inst_id`만 존재한다
- **WHEN** 시스템이 실행 컨텍스트를 초기화한다
- **THEN** 프로세스 인스턴스 식별자는 `proc_inst_id` 값으로 해소된다

### Requirement: 멀티 에이전트 크루 구성 및 실행
시스템은 작업 컨텍스트의 `extras`에 포함된 에이전트 정보, 사용자 정보, 폼 정의, 활동명, 피드백, 소스 자료를 사용하여 멀티 에이전트 크루를 구성하고 SHALL 실행한다. 에이전트 정보가 제공되지 않으면 기본 범용 에이전트로 작업을 수행한다.

#### Scenario: 제공된 에이전트 정보로 크루 실행
- **GIVEN** 작업 컨텍스트의 `extras.agents`에 하나 이상의 에이전트 정보가 있다
- **WHEN** 시스템이 작업을 실행한다
- **THEN** 시스템은 제공된 에이전트들로 크루를 구성하여 작업을 수행한다

#### Scenario: 에이전트 정보 부재 시 기본 에이전트 사용
- **GIVEN** 작업 컨텍스트에 에이전트 정보가 제공되지 않았다
- **WHEN** 시스템이 작업을 실행한다
- **THEN** 시스템은 기본 범용 에이전트를 생성하여 작업을 수행한다

### Requirement: 진행 및 최종 결과 이벤트 발행
시스템은 작업 실행 결과를 이벤트 큐로 SHALL 전달한다. 최종 폼 결과가 있으면 `working` 상태(`event_type: task_started`)와 `completed` 상태(`event_type: task_completed`)의 `TaskStatusUpdateEvent` 쌍을 발행하고, 실행 결과 전체를 담은 `TaskArtifactUpdateEvent` 아티팩트 `crewai_action_result`를 발행한다. 모든 이벤트는 프로세스 인스턴스 식별자와 작업 식별자를 포함한다.

#### Scenario: 폼 결과가 있는 작업의 결과 이벤트 발행
- **GIVEN** 작업 실행으로 비어 있지 않은 폼 결과 데이터가 생성되었다
- **WHEN** 시스템이 결과를 전달한다
- **THEN** 시스템은 `working`(`task_started`)과 `completed`(`task_completed`) 상태 이벤트 쌍을 발행한다
- **AND** `completed` 이벤트 본문에 폼 결과 데이터가 포함된다
- **AND** 시스템은 아티팩트 `crewai_action_result`를 발행한다

#### Scenario: 폼 결과가 비어 있는 작업
- **GIVEN** 작업 실행 결과 폼 데이터가 비어 있다
- **WHEN** 시스템이 결과를 전달한다
- **THEN** 시스템은 폼 결과 이벤트 쌍을 발행하지 않는다
- **AND** 시스템은 아티팩트 `crewai_action_result`는 발행한다

### Requirement: 에이전트 출력 해석 및 폼 데이터 정규화
시스템은 에이전트 실행 결과를 해석하여 `폼_데이터`를 정규화하고, 정규화된 결과를 `form_id` 기준으로 래핑하여 SHALL 제공한다. 코드 펜스로 감싸진 출력, 백틱으로 감싼 값, 줄바꿈으로 연결된 다중 JSON 객체, `result` 키로 감싸진 출력 등 비정상 형식도 가능한 범위에서 복구한다.

#### Scenario: 코드 펜스로 감싸진 출력 복구
- **GIVEN** 에이전트 출력이 코드 펜스(```)로 감싸진 JSON이다
- **WHEN** 시스템이 결과를 해석한다
- **THEN** 시스템은 코드 펜스를 제거하고 폼 데이터를 정규화한다

#### Scenario: 다중 JSON 객체 출력 병합
- **GIVEN** 에이전트 출력이 줄바꿈으로 연결된 여러 JSON 객체로 구성되어 있다
- **WHEN** 시스템이 결과를 해석한다
- **THEN** 시스템은 각 JSON 객체를 파싱하여 하나의 결과로 병합한다

### Requirement: 도구 로딩 실패에 대한 작업 지속성
시스템은 에이전트 도구 로딩이 실패해도 작업을 중단하지 않고 SHALL 계속한다. HTTP/MCP 연결 오류 또는 그 외 오류가 발생하면 해당 에이전트는 도구 없이 실행을 진행한다.

#### Scenario: MCP 도구 로딩 실패 시 도구 없이 진행
- **GIVEN** 에이전트에 할당된 도구의 MCP 연결이 실패한다
- **WHEN** 시스템이 크루를 구성한다
- **THEN** 시스템은 해당 에이전트를 도구 없이 구성하여 작업을 계속한다
- **AND** 작업은 도구 로딩 실패만을 이유로 실패 처리되지 않는다

### Requirement: 실행 자원 정리
시스템은 작업 실행이 성공하든 실패하든 종료 시 MCP 어댑터 자원 정리를 SHALL 시도한다. 정리 중 오류가 발생해도 작업 결과 처리에는 영향을 주지 않는다.

#### Scenario: 작업 종료 시 자원 정리
- **GIVEN** 작업 실행이 완료되거나 오류로 종료되었다
- **WHEN** 시스템이 실행을 마무리한다
- **THEN** 시스템은 MCP 어댑터 정리를 시도한다
- **AND** 정리 중 오류가 발생해도 이미 산출된 결과 처리는 유지된다

### Requirement: 작업 취소 요청 수용
시스템은 작업 취소 요청을 받으면 이를 수용하고 SHALL 정상적으로 반환한다.

#### Scenario: 취소 요청 처리
- **GIVEN** 진행 중이거나 대기 중인 작업이 있다
- **WHEN** 클라이언트가 작업 취소를 요청한다
- **THEN** 시스템은 취소 요청을 오류 없이 수용한다
