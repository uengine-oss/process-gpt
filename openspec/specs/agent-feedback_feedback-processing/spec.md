# 피드백 처리 명세

## Purpose
사용자가 완료된 워크아이템에 남긴 피드백을 시스템이 주기적으로 자동 감지하여, 관련 에이전트별 학습 후보로 변환하고 에이전트 지식으로 반영한 뒤 처리 상태를 갱신하는 능력을 보장한다. 클라이언트는 워크아이템에 피드백을 등록하는 것만으로 별도 호출 없이 에이전트 학습이 진행됨을 기대할 수 있다.

## Requirements

### Requirement: 완료된 워크아이템 피드백 자동 폴링
시스템은 약 7초 간격으로 처리 대상 피드백 작업을 자동으로 한 건씩 선점(claim)하여 처리해야 하며, 피드백이 비어 있지 않고 `proc_def_id`가 존재하며 `feedback_status`가 비었거나 `REQUESTED`인 워크아이템만 대상으로 SHALL 삼는다.

#### Scenario: 처리 대상 피드백 작업 선점
- **GIVEN** 워크아이템에 비어 있지 않은 `feedback` 값과 비어 있지 않은 `proc_def_id`가 있고, `feedback_status`가 비어 있거나 `REQUESTED`이다
- **WHEN** 폴링 주기가 도래한다
- **THEN** 시스템은 해당 워크아이템 한 건을 선점하고 `feedback_status`를 `STARTED`로 변경한다
- **AND** 동시에 실행되는 다른 처리기가 같은 워크아이템을 중복 선점하지 않도록 잠금을 적용한다

#### Scenario: 처리 대상이 없는 경우
- **GIVEN** 조건을 만족하는 피드백 작업과 초기 지식 셋팅 대상이 모두 없다
- **WHEN** 폴링 주기가 도래한다
- **THEN** 시스템은 어떤 작업도 수행하지 않고 대기 상태를 유지한다

#### Scenario: 빈 피드백 또는 proc_def_id 누락 작업 제외
- **GIVEN** 워크아이템의 `feedback`이 `[]` 또는 `{}`이거나 `proc_def_id`가 비어 있다
- **WHEN** 폴링 주기가 도래한다
- **THEN** 시스템은 해당 워크아이템을 처리 대상으로 선점하지 않는다

### Requirement: 피드백 처리 상태 전이
시스템은 선점한 워크아이템의 처리 진행에 따라 `feedback_status`를 관측 가능한 상태로 SHALL 갱신한다. 처리 시작 시 `PROCESSING`, 정상 종료 시 `COMPLETED`, 에이전트 미발견·처리 오류·미커밋 시 `FAILED`로 기록한다.

#### Scenario: 정상 처리 완료
- **GIVEN** 시스템이 피드백 작업을 선점하고 대상 에이전트와 학습 후보를 확보했다
- **WHEN** 모든 학습 후보가 오류 없이 처리된다
- **THEN** 시스템은 워크아이템의 `feedback_status`를 `COMPLETED`로 기록한다

#### Scenario: 매칭된 학습 후보가 없는 경우
- **GIVEN** 시스템이 피드백 작업을 선점하고 대상 에이전트를 찾았다
- **WHEN** 피드백 분석 결과 학습 후보가 하나도 생성되지 않는다
- **THEN** 시스템은 워크아이템의 `feedback_status`를 `COMPLETED`로 기록하고 정상 종료한다

#### Scenario: 일부 에이전트 처리 실패
- **GIVEN** 시스템이 여러 학습 후보를 처리한다
- **WHEN** 하나 이상의 에이전트 처리에서 오류 또는 미커밋이 발생한다
- **THEN** 시스템은 워크아이템의 `feedback_status`를 `FAILED`로 기록한다
- **AND** 폴링은 중단되지 않고 다음 주기를 계속 진행한다

### Requirement: 대상 에이전트 식별
시스템은 워크아이템의 `user_id`로 대상 에이전트를 우선 식별하고, 식별에 실패하면 `assignees` 항목의 `endpoint`에서 에이전트를 탐색해야 하며, 끝내 대상 에이전트를 찾지 못하면 작업을 `FAILED`로 SHALL 종료한다.

#### Scenario: user_id 기반 에이전트 식별
- **GIVEN** 워크아이템의 `user_id`에 쉼표로 구분된 식별자가 있고 해당 에이전트가 존재한다
- **WHEN** 시스템이 대상 에이전트를 조회한다
- **THEN** 시스템은 `user_id`에 해당하는 에이전트들을 처리 대상으로 확보한다

#### Scenario: assignees 기반 대체 식별
- **GIVEN** `user_id`로 대상 에이전트를 하나도 찾지 못했고 `assignees`에 `endpoint` 식별자가 있다
- **WHEN** 시스템이 대상 에이전트를 조회한다
- **THEN** 시스템은 `assignees`의 `endpoint`에 등록된 에이전트를 처리 대상으로 확보한다

#### Scenario: 대상 에이전트 미발견
- **GIVEN** `user_id`와 `assignees` 어느 경로로도 대상 에이전트를 찾지 못했다
- **WHEN** 시스템이 피드백 작업 처리를 시도한다
- **THEN** 시스템은 워크아이템의 `feedback_status`를 `FAILED`로 기록하고 종료한다

### Requirement: 시간순 피드백 통합 학습 후보 생성
시스템은 워크아이템에 누적된 피드백을 시간 순서로 해석하여 가장 최근 피드백을 최우선으로 반영한 에이전트별 학습 후보를 생성해야 하며, 작업 지시사항과 해당 워크아이템의 이벤트 로그를 함께 맥락으로 SHALL 활용한다. 각 학습 후보는 통합된 학습 가이드(`content`)와 의도 힌트(`intent_hint`)를 포함한다.

#### Scenario: 배열 피드백의 최신 항목 우선 반영
- **GIVEN** 워크아이템의 `feedback`이 시간 정보를 가진 여러 피드백 항목의 배열이다
- **WHEN** 시스템이 학습 후보를 생성한다
- **THEN** 시스템은 가장 최근 피드백 내용을 최우선 기준으로 삼아 학습 가이드를 구성한다
- **AND** 이전 피드백과 작업 지시사항, 이벤트 로그는 맥락 정보로 반영한다

#### Scenario: 단순 재시도 요청 처리
- **GIVEN** 피드백 내용이 "다시 시도", "재시도", "try again"과 같은 단순 재시도 요청이다
- **WHEN** 시스템이 해당 피드백을 처리한다
- **THEN** 시스템은 새로운 지식을 저장하지 않고 처리를 종료한다

### Requirement: 폴링 지속성
시스템은 개별 피드백 작업 처리 중 오류가 발생하더라도 폴링 루프를 중단하지 않고 다음 주기를 계속 SHALL 진행한다. 한 폴링 주기에서 피드백 작업 1건과 초기 지식 셋팅 대상 1건을 각각 처리한다.

#### Scenario: 작업 처리 중 예외 발생 시 폴링 유지
- **GIVEN** 폴링이 실행 중이다
- **WHEN** 특정 피드백 작업 처리 중 예외가 발생한다
- **THEN** 시스템은 해당 워크아이템 상태를 `FAILED`로 기록 시도하고 폴링 루프를 계속 유지한다
