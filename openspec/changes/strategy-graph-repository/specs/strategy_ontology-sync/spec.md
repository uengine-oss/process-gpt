# strategy_ontology-sync — 플랫폼 원천 데이터의 온톨로지 인제스천(ETL)

## ADDED Requirements

### Requirement: 원천 데이터 변경분이 온톨로지 그래프에 인제스천된다
strategy 서비스는 플랫폼 원천 데이터가 저장·수정·삭제될 때 그 변경분을 감지해 온톨로지 그래프에 반영해야 한다(SHALL). 변경 감지는 주기적 증분 폴링(`ONTOLOGY_SYNC_INTERVAL_SECONDS`, 기본 60초)으로 수행하며(SHALL), 이미 반영된 데이터를 다시 읽어 중복 노드·관계를 만들어서는 안 된다(MUST NOT). `POST /api/ontology/sync`로 즉시 인제스천을 실행할 수 있어야 하고(SHALL), `full=true` 지정 시 전체를 재구축할 수 있어야 한다(SHALL). 인제스천 결과에는 반영된 노드·관계 수와 마지막 인제스천 시각이 포함되어야 한다(SHALL).

#### Scenario: 신규 원천 데이터의 자동 반영
- **WHEN** 새 프로세스 정의가 저장된 뒤 폴링 주기가 경과하면
- **THEN** 온톨로지 조회에서 해당 프로세스가 `Process` 노드로 나타난다

#### Scenario: 수정분의 반영
- **WHEN** 기존 프로세스 정의의 이름이 수정된 뒤 즉시 인제스천(`POST /api/ontology/sync`)을 호출하면
- **THEN** 온톨로지의 해당 `Process` 노드 이름이 수정된 값으로 갱신되고 노드가 중복 생성되지 않는다

#### Scenario: 삭제분의 반영
- **WHEN** 원천에서 삭제된 프로세스 정의에 대해 인제스천이 수행되면
- **THEN** 해당 `Process` 노드와 그 노드에 연결된 관계가 온톨로지에서 제거된다

### Requirement: 프로세스 정의가 프로세스 레이어와 역할자 관계로 반영된다
프로세스 정의 인제스천은 정의를 `Process` 노드로, 정의에 포함된 활동을 `Task` 노드와 `CONTAINS_TASK` 관계로 반영해야 한다(SHALL). 프로세스 정의에 이미 포함된 활동별 역할자 정보(사용자 또는 에이전트)를 활용해 해당 리소스 노드에서 `Task`로 향하는 `PERFORMS` 관계를 생성해야 한다(SHALL).

#### Scenario: 프로세스 정의의 태스크 분해
- **WHEN** 3개 활동으로 구성된 프로세스 정의가 인제스천되면
- **THEN** 온톨로지에 `Process` 노드 1개와 `CONTAINS_TASK`로 연결된 `Task` 노드 3개가 존재한다

#### Scenario: 정의상 역할자의 수행 관계 생성
- **WHEN** 활동의 역할자로 에이전트가 지정된 프로세스 정의가 인제스천되면
- **THEN** 해당 `Agent` 노드에서 그 활동의 `Task` 노드로 향하는 `PERFORMS` 관계가 존재한다

### Requirement: 사용자·에이전트·조직도가 리소스 레이어로 반영된다
사용자 원천 데이터는 사람과 AI 에이전트를 구분해 각각 `User`, `Agent` 노드로 반영해야 한다(SHALL). 조직도(configuration key='organization')는 조직 단위를 `Team` 노드로, 조직 계층을 `HAS_SUB_TEAM` 관계로, 구성원 소속을 `MEMBER_OF` 관계로 반영해야 한다(SHALL).

#### Scenario: 사람과 에이전트의 구분 반영
- **WHEN** 사람 사용자 1명과 에이전트 1개가 등록된 테넌트가 인제스천되면
- **THEN** 온톨로지에 `User` 노드 1개와 `Agent` 노드 1개가 각각 존재한다

#### Scenario: 조직도 반영
- **WHEN** 상위 팀과 하위 팀, 하위 팀 소속 구성원이 정의된 조직도가 인제스천되면
- **THEN** 상위 `Team`에서 하위 `Team`으로 `HAS_SUB_TEAM` 관계가, 구성원 노드에서 하위 `Team`으로 `MEMBER_OF` 관계가 존재한다

### Requirement: 변경된 스킬은 LLM 분석으로 스킬 간 관계가 추출된다
스킬 원천 데이터는 `Skill` 노드와 에이전트의 `USES_SKILL` 관계로 반영해야 한다(SHALL). 스킬이 새로 저장되거나 수정되면 그 스킬의 본문을 LLM으로 분석해 다른 스킬에 대한 상속/포크 관계(`INHERITS`)와 참조/호출 관계(`REFERENCES`)를 추출하고 온톨로지에 반영해야 한다(SHALL). 추출된 관계는 대상 스킬이 온톨로지에 실제 존재하는 경우에만 생성해야 하며(MUST), 추출 근거를 관계 속성으로 남겨야 한다(SHALL). 변경되지 않은 스킬에 대해 LLM 분석을 반복해서는 안 되고(MUST NOT), LLM이 설정되지 않은 환경에서는 노드 동기화만 수행하고 관계 추출은 보류 상태로 기록해야 한다(SHALL).

#### Scenario: 스킬 참조 관계 추출
- **WHEN** 본문에서 기존 스킬(예: PPTX 생성 스킬)을 호출해 사용하는 새 스킬이 저장되고 인제스천되면
- **THEN** 새 `Skill` 노드에서 기존 `Skill` 노드로 향하는 `REFERENCES` 관계가 추출 근거와 함께 생성된다

#### Scenario: 스킬 상속 관계 추출
- **WHEN** 기존 스킬을 포크해 기능을 확장한 스킬이 저장되고 인제스천되면
- **THEN** 새 `Skill` 노드에서 원본 `Skill` 노드로 향하는 `INHERITS` 관계가 생성된다

#### Scenario: 미변경 스킬의 분석 생략
- **WHEN** 스킬 변경 없이 인제스천이 반복 실행되면
- **THEN** 스킬에 대한 LLM 분석이 호출되지 않고 기존 관계가 유지된다

#### Scenario: LLM 미설정 환경
- **WHEN** LLM 설정 없이 새 스킬이 인제스천되면
- **THEN** `Skill` 노드는 생성되고, 관계 추출이 보류되었음이 인제스천 결과에 나타난다

### Requirement: 수행 이력이 수행 관계의 실행 지표로 집계된다
완료된 워크아이템 이력을 증분 집계해 수행자(사용자/에이전트)와 태스크 간 `PERFORMS` 관계의 실행 지표(수행 건수, 평균 처리시간)를 갱신해야 한다(SHALL).

#### Scenario: 실행 지표 집계
- **WHEN** 특정 에이전트가 동일 태스크의 워크아이템을 2건 완료한 뒤 인제스천이 수행되면
- **THEN** 그 에이전트와 태스크 간 `PERFORMS` 관계의 수행 건수가 2로, 평균 처리시간이 두 건의 평균으로 조회된다
