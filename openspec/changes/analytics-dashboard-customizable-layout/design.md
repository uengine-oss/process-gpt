## Context

분석 대시보드 화면(`/analytics`)은 Vue 3 + Vuetify + Pinia 기반 프론트엔드에 고정 배치로 구현되어 있다. 위젯(통계 카드, 작업 분포 도넛, 월별 추이, FTE 히트맵, 부서별 작업량, Agent-대-Human, 프로세스 성능, 병목 분석, 최근 인스턴스 표)은 항상 같은 순서·크기로 렌더된다. 위젯 데이터는 두 경로에서 온다: 대부분은 게이트웨이 `/api/analytics/*` 를 통해 백엔드 분석 서비스(`process-gpt-analytic`)에서, 일부(Domain별 Sub Process, 조직도)는 프론트엔드가 Supabase(`configuration`, `departments`)를 직접 조회한다.

제약:
- 백엔드 분석 서비스(`process-gpt-analytic`)는 이 저장소에 소스가 없는 사전 빌드 이미지(`ghcr.io/uengine-oss/process-gpt-analytic`)로, 이번 변경에서 신규 엔드포인트를 추가하기 어렵다.
- 프론트엔드는 이미 Supabase에 직접 읽기/쓰기를 수행하는 패턴을 갖고 있다.
- 현재 의존성에는 리오더용 `vuedraggable`은 있으나 2D 자유 배치 + 리사이즈가 가능한 그리드 레이아웃 라이브러리는 없다.

## Goals / Non-Goals

**Goals:**
- 사용자가 분석 대시보드 위젯을 드래그로 이동하고, 크기를 조정하고, 숨기거나 다시 추가할 수 있게 한다.
- 사용자별 레이아웃을 영속화하고 재접속 시 복원한다(테넌트·사용자 단위 격리).
- 위젯 종류가 추가/폐기되어도 저장된 레이아웃이 안전하게 폴백·하위 호환되도록 한다.
- 위젯의 데이터 조회 로직은 그대로 두고, 배치/표시만 개인화 계층으로 감싼다.

**Non-Goals:**
- 위젯이 표시하는 지표의 계산 로직, 분석용 뷰/테이블 스키마, `/api/analytics/*` 데이터 엔드포인트 변경.
- 위젯 내부 상호작용(드릴다운, 필터)이나 신규 지표 위젯 추가.
- 조직/역할 단위 공유 레이아웃, 관리자 강제 레이아웃(개인 레이아웃만 다룬다).

## Decisions

### 결정 1: 그리드 엔진으로 Vue 3용 grid-layout 라이브러리 도입
- **선택**: `grid-layout-plus`(Vue 3용 vue-grid-layout 후속) 같은 반응형 드래그·리사이즈 그리드 라이브러리를 도입해 위젯을 그리드 항목으로 감싼다. 각 위젯은 `{i, x, y, w, h, minW, minH}` 로 표현한다.
- **대안**: 이미 있는 `vuedraggable`(Sortable 기반) 재사용. 리오더는 가능하지만 2D 자유 배치와 리사이즈를 지원하지 않아 요구사항(위치 이동 + 크기 조정)을 충족하지 못한다.
- **근거**: 스펙이 위치 이동과 크기 조정을 모두 요구하므로 그리드+리사이즈를 기본 제공하는 라이브러리가 필요하다. 조회 모드에서는 드래그·리사이즈를 비활성화하여 기존 조회 경험을 보존한다.

### 결정 2: 위젯 레지스트리(카탈로그)로 위젯 정의를 중앙화
- **선택**: 각 위젯을 `widgetId`(안정적 문자열 키), 제목, 렌더 컴포넌트, 기본 `{x,y,w,h}`, `minW/minH`, 기본 표시 여부를 갖는 레지스트리로 정의한다. 대시보드는 "레지스트리 × 사용자 레이아웃"을 병합해 렌더한다.
- **대안**: 위젯을 템플릿에 하드코딩하고 배치만 배열로 관리. 신규/폐기 위젯 처리와 폴백이 어렵다.
- **근거**: 저장된 레이아웃은 `widgetId`만 참조하므로, 레지스트리에 없는 `widgetId`는 무시하고 레이아웃에 없는 신규 위젯은 기본 위치로 노출하는 하위 호환·폴백 규칙을 단순하게 구현할 수 있다.

### 결정 3: 레이아웃 영속화는 프론트엔드 Supabase 직접 접근으로 처리
- **선택**: Supabase에 사용자별 레이아웃 저장용 테이블(예: `user_dashboard_layout`)을 신설하고, 프론트엔드가 기존 Supabase 접근 패턴으로 저장/조회한다. 키는 `(tenant_id, user_id, dashboard_key)`, 본문은 위젯 배치 문서(JSON)와 스키마 버전.
- **대안 A**: `process-gpt-analytic` 에 레이아웃 저장/조회 엔드포인트 추가 — 소스가 저장소에 없어 이번 범위에서 구현 불가.
- **대안 B**: 기존 `configuration` 테이블에 per-user 키로 저장 — `configuration` 은 테넌트/전역 설정용이라 사용자 단위 격리와 RLS 표현이 부자연스럽다.
- **근거**: 프론트엔드는 이미 Supabase 직접 접근 패턴을 갖고 있고, RLS로 테넌트·사용자 격리를 명확히 표현할 수 있어 계약(저장/복원/격리)을 가장 단순하게 만족한다. 마이그레이션은 기존 분석 마이그레이션과 같은 위치(`services/frontend/supabase/migrations/`)에 둔다.

### 결정 4: 편집 상태는 Pinia 스토어로 관리하고 조회 모드를 기본값으로
- **선택**: 편집 모드 on/off, 현재 레이아웃(드래프트/저장본), 숨김 위젯 목록을 Pinia 스토어에 두고, 저장 시에만 Supabase에 반영한다. 초기화는 개인 레이아웃 레코드를 삭제하고 기본 레이아웃으로 되돌린다.
- **근거**: 편집 중 드래프트와 저장본을 분리해, 저장 전 취소/이탈 시 마지막 저장 상태를 보존한다.

## Risks / Trade-offs

- [신규 그리드 라이브러리 도입으로 번들 크기 증가/Vuetify 레이아웃과 충돌 가능] → 대시보드 화면에 한정해 지연 로드하고, 위젯을 Vuetify 카드로 감싼 상태에서 그리드 항목만 라이브러리가 관리하도록 격리한다.
- [저장된 레이아웃과 위젯 레지스트리 불일치로 빈 화면 위험] → 로드 시 레지스트리 기준으로 병합·정규화하고, 파싱 불가/누락 시 기본 레이아웃으로 폴백하는 것을 스펙 요구사항으로 못박는다.
- [반응형(모바일/좁은 화면)에서 자유 배치가 깨질 수 있음] → 좁은 브레이크포인트에서는 단일 열로 자동 축소(리사이즈/자유 이동 제한)하고, 저장은 기본(넓은) 브레이크포인트 기준으로 한다.
- [RLS 미설정 시 사용자 간 레이아웃 노출] → 테이블 신설과 동시에 `(tenant_id, user_id)` 기준 RLS 정책을 마이그레이션에 포함하고 격리 시나리오를 E2E로 검증한다.
- [analytic-service에 저장 API를 두지 않아 백엔드 일관성 저하] → 레이아웃은 순수 프론트엔드 개인화 데이터이며 지표 계산과 무관하므로 Supabase 직접 접근으로 분리 보관하는 것이 합리적. 향후 백엔드 노출이 필요하면 같은 테이블을 소스로 엔드포인트를 추가할 수 있다.

## Migration Plan

1. Supabase 마이그레이션 추가: `user_dashboard_layout` 테이블 + `(tenant_id, user_id)` RLS 정책. 기존 사용자에게는 레코드가 없으므로 기본 레이아웃 폴백으로 자연 동작한다(데이터 백필 불필요).
2. 프론트엔드에 그리드 라이브러리 의존성 추가 및 위젯 레지스트리 도입, `Dashboard.vue` 를 레지스트리+레이아웃 병합 렌더로 재구성.
3. 편집 모드/저장/초기화 UI 및 Pinia 스토어, Supabase 저장·조회 연동.
4. 롤백: 그리드/편집 기능은 기능 플래그(또는 편집 토글 비노출)로 끌 수 있게 하고, 문제가 있으면 기본 고정 레이아웃 렌더로 되돌린다. `user_dashboard_layout` 테이블은 조회되지 않아도 기존 화면에 영향 없다.

## Open Questions

- 레이아웃 저장 트리거: 편집 종료 시 자동 저장할지, 명시적 "저장" 버튼을 둘지(초안은 명시적 저장 + 취소).
- 반응형 브레이크포인트별로 별도 레이아웃을 저장할지, 단일 레이아웃을 화면 폭에 맞춰 축소만 할지(초안은 단일 저장 + 축소).
- 향후 조직/역할 단위 기본 레이아웃(관리자 지정)을 도입할지 — 이번 범위 밖이나 테이블 설계 시 확장 여지를 남긴다.
